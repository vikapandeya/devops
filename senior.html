A. DevSecOps — Security in the Pipeline

Security⏱ ~50 min
⚠ Security is not an afterthought — it's a first-class citizen in every stage of the pipeline.

Secure CI/CD Pipeline — Shift Left Security

Code Push → SAST (SonarQube) → Secret Scan (TruffleHog) → Build → SCA (Snyk/Trivy) → Container Scan (Trivy) → Sign Image (Cosign) → Deploy → DAST (OWASP ZAP) → Runtime Security (Falco) → Monitor

GitHub Actions — Security Scanning Pipeline

# .github/workflows/security.yml
name: Security Scan

on: [push, pull_request]

jobs:
  sast:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: SonarQube scan
        uses: sonarqube/sonarqube-scan-action@v2
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
          SONAR_HOST_URL: ${{ secrets.SONAR_HOST_URL }}

  secrets:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }
      - name: Scan for secrets
        uses: trufflesecurity/trufflehog@main
        with:
          path: ./
          base: ${{ github.event.repository.default_branch }}

  container-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build image
        run: docker build -t myapp:${{ github.sha }} .
      - name: Run Trivy vulnerability scanner
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: myapp:${{ github.sha }}
          format: sarif
          output: trivy-results.sarif
          severity: CRITICAL,HIGH
          exit-code: '1'         # fail if CRITICAL/HIGH found
      - name: Upload results
        uses: github/codeql-action/upload-sarif@v3
        with:
          sarif_file: trivy-results.sarif

  sign-image:
    needs: container-scan
    runs-on: ubuntu-latest
    steps:
      - name: Sign container image with Cosign
        run: cosign sign --yes myregistry/myapp:${{ github.sha }}
        env:
          COSIGN_EXPERIMENTAL: 1
yaml

Error Budget Remaining Action Required ───────────────────────────────────────────────── > 50% Normal operations, can ship new features freely 25–50% Review release velocity, increase testing 10–25% Freeze non-critical releases, focus on reliability < 10% All hands on reliability, no new features 0% (burnt) Freeze all deployments until next budget period

Incident Management Runbook

# Incident Response Phases

# 1. DETECT (automated)
#    Prometheus alert fires → PagerDuty page → On-call engineer notified

# 2. RESPOND (< 5 min)
kubectl get pods -n production               # check pod health
kubectl top pods -n production               # resource spike?
kubectl get events -n production --sort-by='.lastTimestamp' | tail -20
kubectl logs deployment/webapp --since=5m   # recent logs

# 3. MITIGATE (< 30 min)
# Rollback if recent deploy caused it
kubectl rollout undo deployment/webapp -n production
kubectl rollout status deployment/webapp -n production

# Scale out if traffic spike
kubectl scale deployment/webapp --replicas=10 -n production

# Circuit breaker — divert traffic to maintenance page
kubectl patch ingress webapp-ingress -n production \
  -p '{"spec":{"rules":[{"host":"app.com","http":{"paths":[{"path":"/","backend":{"service":{"name":"maintenance-svc","port":{"number":80}}}}]}}]}}'

# 4. RESOLVE — fix root cause
# 5. POSTMORTEM — blameless, within 24-48 hours after incident

CAP Theorem for DevOps

📚 CAP: you can only guarantee 2 of 3: Consistency, Availability, Partition Tolerance. In distributed systems, partition tolerance is mandatory (networks fail), so you choose C or A.
SystemChoosesTrade-off
PostgreSQL, MySQLCPWill block writes during network partition rather than serve stale data
Cassandra, DynamoDBAPAlways available, may return slightly stale data
Redis (standalone)CPReturns error if primary unreachable
Redis ClusterAPServes reads from replicas even if primary fails

Database Scaling Strategies

# Read Replica — for read-heavy workloads
Primary (writes) ──replication──► Replica 1 (reads)
                              ──► Replica 2 (reads)
                              ──► Replica 3 (reads — analytics)

# Sharding — horizontal partitioning
Shard 1: users A-M (user_id % 3 == 0)
Shard 2: users N-R (user_id % 3 == 1)
Shard 3: users S-Z (user_id % 3 == 2)

# CQRS — separate read/write models
Write API → Command → Event Store → Projector → Read DB
                                              → Read API → Client

# ADR-0001: Use Kubernetes Over ECS for Container Orchestration

## Status: Accepted (2025-01-15)

## Context
We need to orchestrate 30+ microservices across 3 environments.
Our team has K8s expertise. We need custom scheduling, service mesh, and GitOps.

## Decision
We will use EKS (managed Kubernetes) over ECS.

## Consequences

### Positive
- Portable — can migrate to other clouds or on-prem if needed
- Rich ecosystem: Helm, ArgoCD, Karpenter, Istio, KEDA
- More control over networking and scheduling
- Team expertise match

### Negative
- Higher operational complexity than ECS
- EKS control plane costs ~$73/month per cluster
- More configuration required initially

## Alternatives Considered
- **ECS Fargate**: Simpler, lower cost, but vendor lock-in and less ecosystem
- **Nomad**: Great for multi-cloud but smaller community
- **Docker Compose on EC2**: Too manual, doesn't scale

Senior Interview Prep

Interview⏱ ~40 min
How do you design a system that can handle 1 million requests per day? ▼
1M req/day = ~12 req/sec average, but traffic is spiky — plan for 10x peak = 120 req/sec. Architecture: CDN (CloudFront) for static assets reduces origin load 70-80%. Load balancer (ALB) distributes across 3+ app servers in 2 AZs. Horizontal scaling via ASG — scale out at 70% CPU. Redis for session caching (reduce DB hits). Read replicas for DB (separate read/write). Connection pooling (PgBouncer) at 25 connections per app server. In K8s: 3 replicas baseline, HPA scales to 20 at peak. Monitor P99 latency, error rate, and connection pool exhaustion.
How do you implement GitOps in practice? ▼
GitOps means Git is the single source of truth for infrastructure and application state. Implementation with ArgoCD: 1) Separate "config repo" (Helm charts, K8s manifests) from "app repo" (code). 2) CI pipeline builds image, pushes to registry, then updates the image tag in the config repo. 3) ArgoCD watches the config repo and automatically syncs cluster state to match Git. 4) Production changes require PR review — no kubectl apply by humans in production. 5) Rollback = git revert → ArgoCD syncs. Benefits: full audit trail, anyone can see current state, rollbacks are reliable, no config drift. Use ApplicationSet for multi-cluster/multi-env management.
Your production database is at 95% disk. Walk me through how you handle it. ▼
Immediate (0-30 min): 1) Alert triage — confirm free space trend with df -h and growth rate. 2) Find largest tables: SELECT relname, pg_size_pretty(pg_total_relation_size(relid)) FROM pg_stat_user_tables ORDER BY pg_total_relation_size(relid) DESC. 3) Clear safe large files — WAL archives, temp tables, old logs. 4) If RDS: modify storage for online autoscaling (adds storage live). Short-term: archive/delete old data with ROW-level delete, not table drop. Run VACUUM to reclaim space. Long-term: Implement data retention policy. Partition large tables by date. Move audit logs to S3 + Athena. Set CloudWatch alarm at 70% to give warning. Consider S3-backed columnar storage (Redshift) for analytics data.
What is your approach to incident postmortems? ▼
Blameless postmortem within 24-48 hours. Structure: 1) Timeline: What happened, when detected, who involved, what actions were taken. 2) Root cause: 5 Whys analysis — keep asking WHY until you reach a systemic issue (not "human error"). 3) Contributing factors: Monitoring gaps, deployment without testing, missing runbooks. 4) Action items: Each has an owner and deadline. Categories: detection improvement (shorter MTTR), prevention (tests, validation), process (runbooks, rollback procedures). 5) Share widely — make postmortems accessible to all engineers, not just the team involved. Normalize incidents as learning opportunities. Track action item completion. Never assign blame to individuals.
How do you explain technical trade-offs to non-technical stakeholders? ▼
Frame everything in business terms, not technical terms. Instead of "we need to refactor the auth service to reduce technical debt" say "our login system is slowing down 3x a year for maintenance — investing 2 sprints now will save 6 sprint-weeks over the next year." Use the RICE framework: Reach (how many users affected?), Impact (severity?), Confidence (how sure?), Effort (person-weeks). Create a visual — a simple risk/effort matrix on a single slide. Always present options, not just one solution: Option A (fast/cheap/risky) vs Option B (slower/costly/reliable). Never hide risks — stakeholders trust engineers who proactively flag risks over those who promise certainty. End with a clear recommendation and the key assumption that makes it valid.
How do you set up a multi-cluster Kubernetes strategy? ▼
Reasons for multi-cluster: blast radius isolation, compliance (PCI/HIPAA data isolation), multi-region availability, dev/staging/prod separation. Strategies: 1) Fleet management: ArgoCD ApplicationSet or Fleet (Rancher) to manage 100s of clusters from single control plane. 2) Service discovery: Istio multi-cluster (shared control plane) or Skupper for cross-cluster service exposure. 3) DNS routing: Route53 failover between regional clusters. 4) Shared infrastructure: Central ECR, shared VPC peering or AWS Transit Gateway. 5) GitOps: Cluster-specific Helm value files in Git, ArgoCD syncs appropriate cluster. Key challenge: data gravity — keep services close to their databases to avoid cross-AZ/cross-region latency costs.

Scenario-Based Questions (Answer Framework)

📋 For scenario questions, use the STAR format: Situation, Task, Action, Result. Always quantify the impact.
  • → "Tell me about a time you improved system reliability" — discuss SLO improvement, MTTR reduction
  • → "How did you handle a major production incident?" — timeline, communication, blameless review
  • → "Describe a time you convinced your team to adopt a new tool" — data-driven, proof of concept, phased rollout
  • → "How did you reduce cloud costs?" — specific %, tools used (Cost Explorer, Compute Optimizer), initiatives
