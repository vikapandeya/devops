A. DevSecOps — Security in the Pipeline

Security⏱ ~50 min
⚠ Security is not an afterthought — it's a first-class citizen in every stage of the pipeline.

Secure CI/CD Pipeline — Shift Left Security

Code Push → SAST (SonarQube) → Secret Scan (TruffleHog) → Build → SCA (Snyk/Trivy) → Container Scan (Trivy) → Sign Image (Cosign) → Deploy → DAST (OWASP ZAP) → Runtime Security (Falco) → Monitor

GitHub Actions — Security Scanning Pipeline

# .github/workflows/security.yml
name: Security Scan

on: [push, pull_request]

jobs:
  sast:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: SonarQube scan
        uses: sonarqube/sonarqube-scan-action@v2
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
          SONAR_HOST_URL: ${{ secrets.SONAR_HOST_URL }}

  secrets:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }
      - name: Scan for secrets
        uses: trufflesecurity/trufflehog@main
        with:
          path: ./
          base: ${{ github.event.repository.default_branch }}

  container-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build image
        run: docker build -t myapp:${{ github.sha }} .
      - name: Run Trivy vulnerability scanner
        uses: aquasecurity/trivy-action@master
        with:
          image-ref: myapp:${{ github.sha }}
          format: sarif
          output: trivy-results.sarif
          severity: CRITICAL,HIGH
          exit-code: '1'         # fail if CRITICAL/HIGH found
      - name: Upload results
        uses: github/codeql-action/upload-sarif@v3
        with:
          sarif_file: trivy-results.sarif

  sign-image:
    needs: container-scan
    runs-on: ubuntu-latest
    steps:
      - name: Sign container image with Cosign
        run: cosign sign --yes myregistry/myapp:${{ github.sha }}
        env:
          COSIGN_EXPERIMENTAL: 1
yaml

# OPA Rego policy: deny containers running as root

  input.kind == "Deployment"
  container := input.spec.template.spec.containers[_]

SRE Key Concepts

TermDefinitionExample
SLIService Level Indicator — what you measureRequest success rate (%), P99 latency
SLOService Level Objective — the target99.9% success rate, P99 < 500ms
SLAService Level Agreement — the contract99.5% uptime or credit refund
Error BudgetAllowed failure = 100% - SLO0.1% = 43.8 min/month of downtime
ToilManual, repetitive, automatable workManual deploys, ticket-based scaling
MTTRMean Time To Recover from incidentAvg time from alert to resolution
MTBFMean Time Between FailuresAvg time between incidents

Error Budget Policy

Error Budget Remaining Action Required ───────────────────────────────────────────────── > 50% Normal operations, can ship new features freely 25–50% Review release velocity, increase testing 10–25% Freeze non-critical releases, focus on reliability < 10% All hands on reliability, no new features 0% (burnt) Freeze all deployments until next budget period

Incident Management Runbook

# Incident Response Phases

# 1. DETECT (automated)
#    Prometheus alert fires → PagerDuty page → On-call engineer notified

# 2. RESPOND (< 5 min)
kubectl get pods -n production               # check pod health
kubectl top pods -n production               # resource spike?
kubectl get events -n production --sort-by='.lastTimestamp' | tail -20
kubectl logs deployment/webapp --since=5m   # recent logs

# 3. MITIGATE (< 30 min)
# Rollback if recent deploy caused it
kubectl rollout undo deployment/webapp -n production
kubectl rollout status deployment/webapp -n production

# Scale out if traffic spike
kubectl scale deployment/webapp --replicas=10 -n production

# Circuit breaker — divert traffic to maintenance page
kubectl patch ingress webapp-ingress -n production \
  -p '{"spec":{"rules":[{"host":"app.com","http":{"paths":[{"path":"/","backend":{"service":{"name":"maintenance-svc","port":{"number":80}}}}]}}]}}'

# 4. RESOLVE — fix root cause
# 5. POSTMORTEM — blameless, within 24-48 hours after incident

Golden Signals (4 SRE Metrics to Monitor)

⚡

Latency

Time to serve a request. Track P50, P95, P99 — not just average.

📈

Traffic

How many requests per second. Helps understand load patterns.

❌

Errors

Rate of failed requests. 5xx per minute, not absolute numbers.

💾

Saturation

How "full" the service is. CPU queue, memory pressure, disk IO.

Error Budget Management

# Error Budget Policy — what happens when budget is consumed

# SLO: 99.9% availability → error budget: 43.8 min/month

# Budget > 50% remaining:
#   - Ship features normally
#   - Any velocity acceptable

# Budget 10%-50% remaining:
#   - Review risky deployments
#   - Require post-change monitoring window

# Budget < 10% remaining:
#   - Feature freeze — reliability work only
#   - All changes require on-call approval
#   - SRE team can veto deploys

# Budget exhausted:
#   - Full stop on features
#   - Incident retrospective required
#   - Root cause and remediation before resuming

# Toil: repetitive, manual, automatable, tactical work
# Target: < 50% of SRE time on toil
# Measure: track toil hours per week per incident type
# Reduce by: automation, runbooks, self-service tooling

PatternTraffic SplitRollbackCost

Canary5% → 25% → 100%Fast (redirect traffic)Small overhead
Feature FlagsPer-user controlToggle off flagMinimal

CAP Theorem for DevOps

📚 CAP: you can only guarantee 2 of 3: Consistency, Availability, Partition Tolerance. In distributed systems, partition tolerance is mandatory (networks fail), so you choose C or A.
SystemChoosesTrade-off
PostgreSQL, MySQLCPWill block writes during network partition rather than serve stale data
Cassandra, DynamoDBAPAlways available, may return slightly stale data
Redis (standalone)CPReturns error if primary unreachable
Redis ClusterAPServes reads from replicas even if primary fails

Database Scaling Strategies

# Read Replica — for read-heavy workloads
Primary (writes) ──replication──► Replica 1 (reads)
                              ──► Replica 2 (reads)
                              ──► Replica 3 (reads — analytics)

# Sharding — horizontal partitioning
Shard 1: users A-M (user_id % 3 == 0)
Shard 2: users N-R (user_id % 3 == 1)
Shard 3: users S-Z (user_id % 3 == 2)

# CQRS — separate read/write models
Write API → Command → Event Store → Projector → Read DB
                                              → Read API → Client

Node Affinity, Taints & Tolerations

# ── Taint a node (repel pods) ──
kubectl taint nodes gpu-node hardware=gpu:NoSchedule
kubectl taint nodes gpu-node hardware=gpu:NoSchedule-   # remove taint

# ── Toleration (allow pod on tainted node) ──
tolerations:
- key: "hardware"
  operator: "Equal"
  value: "gpu"
  effect: "NoSchedule"

# ── Node Affinity (prefer or require specific nodes) ──
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: node.kubernetes.io/instance-type
          operator: In
          values: ["m5.xlarge", "m5.2xlarge"]
    preferredDuringSchedulingIgnoredDuringExecution:
    - weight: 80
      preference:
        matchExpressions:
        - key: topology.kubernetes.io/zone
          operator: In
          values: ["us-east-1a"]

# ── Pod Anti-Affinity (spread across nodes) ──
affinity:
  podAntiAffinity:
    preferredDuringSchedulingIgnoredDuringExecution:
    - weight: 100
      podAffinityTerm:
        labelSelector:
          matchLabels: { app: webapp }
        topologyKey: kubernetes.io/hostname

# ── PodDisruptionBudget ── guarantee min pods during node drain ──
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: webapp-pdb
spec:
  minAvailable: 2              # always keep at least 2 pods
  # maxUnavailable: 1          # OR allow max 1 down at once
  selector:
    matchLabels: { app: webapp }

OTel Architecture

App (instrumented with OTel SDK) │ traces + metrics + logs ▼ OTel Collector (sidecar or daemonset) │ receive → process → export ├──► Jaeger (traces) ├──► Prometheus (metrics) └──► Loki / Elasticsearch (logs) No vendor lock-in — swap backends without changing app code

# ADR-0001: Use Kubernetes Over ECS for Container Orchestration

## Status: Accepted (2025-01-15)

## Context
We need to orchestrate 30+ microservices across 3 environments.
Our team has K8s expertise. We need custom scheduling, service mesh, and GitOps.

## Decision
We will use EKS (managed Kubernetes) over ECS.

## Consequences

### Positive
- Portable — can migrate to other clouds or on-prem if needed
- Rich ecosystem: Helm, ArgoCD, Karpenter, Istio, KEDA
- More control over networking and scheduling
- Team expertise match

### Negative
- Higher operational complexity than ECS
- EKS control plane costs ~$73/month per cluster
- More configuration required initially

## Alternatives Considered
- **ECS Fargate**: Simpler, lower cost, but vendor lock-in and less ecosystem
- **Nomad**: Great for multi-cloud but smaller community
- **Docker Compose on EC2**: Too manual, doesn't scale

## Review Date: 2026-01-15

# Partition = ordered, immutable sequence (parallelism unit)
# Consumer Group = multiple consumers sharing partitions
# Offset   = position in partition (consumer tracks this)

producer.send('order-created', {

# ── Patterns ──
# Saga Pattern: distributed transactions via events
# Order Service → emits OrderCreated
# → Payment Service consumes, emits PaymentProcessed or PaymentFailed
# → Inventory Service consumes PaymentProcessed, emits InventoryReserved
# → If any fails: compensating transactions reverse prior steps

# CQRS: Command Query Responsibility Segregation
# Commands (write) → Command Handler → Write DB (normalized)
# Queries  (read)  → Query Handler  → Read DB (denormalized/cache)
# Sync via events: write DB emits events → read DB subscribes + updates

# Outbox Pattern (guaranteed event delivery)
# 1. Within same DB transaction: write business record + event to outbox table
# 2. Outbox poller reads unprocessed events
# 3. Publishes to Kafka → marks as published
# Guarantees: no event lost even if app crashes between write + publish

# Title: Use Kafka for Inter-Service Event Streaming
# Date: 2024-03-15
# Status: Accepted

## Context
# Our microservices need reliable async communication.
# Current REST polling causes tight coupling and high latency.
# We need replay capability and multiple consumers per event.

## Decision
# Use Apache Kafka (via AWS MSK) as our event streaming platform.
# Each domain owns its topics. Events are the API contract.

## Consequences
# + Decoupled services, independent scaling
# + Replay events for new consumers or debugging
# + High throughput (millions of events/sec)
# - Operational complexity (broker management, schema evolution)

# - Learning curve for team members unfamiliar with Kafka

## Alternatives Considered
# RabbitMQ: simpler but lacks replay, lower throughput
# AWS SNS/SQS: AWS-only, no replay on SQS, limited ordering

class DecimalEncoder(json.JSONEncoder):
    def default(self, o):
        if isinstance(o, Decimal): return float(o)
        return super().default(o)

def handler(event, context):
    try:
        result = table.scan(Limit=100)
        return {
            "statusCode": 200,
            "headers": {"Content-Type": "application/json"},
            "body": json.dumps(result["Items"], cls=DecimalEncoder)

Senior Interview Prep

Interview⏱ ~40 min
How do you design a system that can handle 1 million requests per day? ▼
1M req/day = ~12 req/sec average, but traffic is spiky — plan for 10x peak = 120 req/sec. Architecture: CDN (CloudFront) for static assets reduces origin load 70-80%. Load balancer (ALB) distributes across 3+ app servers in 2 AZs. Horizontal scaling via ASG — scale out at 70% CPU. Redis for session caching (reduce DB hits). Read replicas for DB (separate read/write). Connection pooling (PgBouncer) at 25 connections per app server. In K8s: 3 replicas baseline, HPA scales to 20 at peak. Monitor P99 latency, error rate, and connection pool exhaustion.
How do you implement GitOps in practice? ▼
GitOps means Git is the single source of truth for infrastructure and application state. Implementation with ArgoCD: 1) Separate "config repo" (Helm charts, K8s manifests) from "app repo" (code). 2) CI pipeline builds image, pushes to registry, then updates the image tag in the config repo. 3) ArgoCD watches the config repo and automatically syncs cluster state to match Git. 4) Production changes require PR review — no kubectl apply by humans in production. 5) Rollback = git revert → ArgoCD syncs. Benefits: full audit trail, anyone can see current state, rollbacks are reliable, no config drift. Use ApplicationSet for multi-cluster/multi-env management.
Your production database is at 95% disk. Walk me through how you handle it. ▼
Immediate (0-30 min): 1) Alert triage — confirm free space trend with df -h and growth rate. 2) Find largest tables: SELECT relname, pg_size_pretty(pg_total_relation_size(relid)) FROM pg_stat_user_tables ORDER BY pg_total_relation_size(relid) DESC. 3) Clear safe large files — WAL archives, temp tables, old logs. 4) If RDS: modify storage for online autoscaling (adds storage live). Short-term: archive/delete old data with ROW-level delete, not table drop. Run VACUUM to reclaim space. Long-term: Implement data retention policy. Partition large tables by date. Move audit logs to S3 + Athena. Set CloudWatch alarm at 70% to give warning. Consider S3-backed columnar storage (Redshift) for analytics data.
What is your approach to incident postmortems? ▼
Blameless postmortem within 24-48 hours. Structure: 1) Timeline: What happened, when detected, who involved, what actions were taken. 2) Root cause: 5 Whys analysis — keep asking WHY until you reach a systemic issue (not "human error"). 3) Contributing factors: Monitoring gaps, deployment without testing, missing runbooks. 4) Action items: Each has an owner and deadline. Categories: detection improvement (shorter MTTR), prevention (tests, validation), process (runbooks, rollback procedures). 5) Share widely — make postmortems accessible to all engineers, not just the team involved. Normalize incidents as learning opportunities. Track action item completion. Never assign blame to individuals.
How do you explain technical trade-offs to non-technical stakeholders? ▼
Frame everything in business terms, not technical terms. Instead of "we need to refactor the auth service to reduce technical debt" say "our login system is slowing down 3x a year for maintenance — investing 2 sprints now will save 6 sprint-weeks over the next year." Use the RICE framework: Reach (how many users affected?), Impact (severity?), Confidence (how sure?), Effort (person-weeks). Create a visual — a simple risk/effort matrix on a single slide. Always present options, not just one solution: Option A (fast/cheap/risky) vs Option B (slower/costly/reliable). Never hide risks — stakeholders trust engineers who proactively flag risks over those who promise certainty. End with a clear recommendation and the key assumption that makes it valid.
How do you set up a multi-cluster Kubernetes strategy? ▼
Reasons for multi-cluster: blast radius isolation, compliance (PCI/HIPAA data isolation), multi-region availability, dev/staging/prod separation. Strategies: 1) Fleet management: ArgoCD ApplicationSet or Fleet (Rancher) to manage 100s of clusters from single control plane. 2) Service discovery: Istio multi-cluster (shared control plane) or Skupper for cross-cluster service exposure. 3) DNS routing: Route53 failover between regional clusters. 4) Shared infrastructure: Central ECR, shared VPC peering or AWS Transit Gateway. 5) GitOps: Cluster-specific Helm value files in Git, ArgoCD syncs appropriate cluster. Key challenge: data gravity — keep services close to their databases to avoid cross-AZ/cross-region latency costs.

How do you design for high availability across multiple AWS regions? ▼
Multi-region HA requires: 1) Active-Active or Active-Passive decision — active-active serves traffic from both regions simultaneously (lower latency globally) while active-passive has a standby that takes over on failure. 2) Route53 with health checks — latency-based or failover routing with endpoint health monitoring. 3) Data replication — Aurora Global Database (read across regions, <1s RPO), DynamoDB Global Tables, or S3 Cross-Region Replication. 4) RTO/RPO targets drive architecture: RPO=0 needs synchronous replication (expensive, higher latency); RPO=5min allows async. 5) Traffic shifting — weighted Route53 records allow gradual cutover. Key challenge: distributed transactions and data consistency across regions require careful design (eventual consistency is the practical answer).
Walk me through designing a zero-trust security architecture ▼
Zero-trust: "never trust, always verify" — no implicit trust based on network location. Key components: 1) Identity-based access — every service has an identity (SPIFFE/SPIRE or AWS IAM role). mTLS everywhere via service mesh (Istio). 2) Least-privilege IAM — workload identity, no static credentials, short-lived tokens via IRSA (IAM Roles for Service Accounts) in EKS. 3) Network micro-segmentation — K8s NetworkPolicy, AWS Security Groups with specific rules, no broad "allow all" within VPC. 4) Secrets management — no secrets in env vars, environment variables sourced from Secrets Manager/Vault at runtime. 5) Audit everything — CloudTrail, K8s audit logs, all API calls logged. 6) Certificate rotation — automated with cert-manager, short TTL certs. Implementation: start with service mesh for mTLS, add OPA policies, then migrate to workload identity.
How do you approach capacity planning for a new service? ▼
Capacity planning process: 1) Define SLOs first — latency targets (p95, p99), availability %, throughput requirements. 2) Load test to find limits — use k6, Locust, or JMeter to ramp up to breaking point. Identify bottleneck (CPU, memory, DB connections, I/O). 3) Model growth — expected RPS growth over 6/12 months based on business projections. 4) Right-size with HPA — set requests to actual p50 usage, limits at p99. Configure HPA to scale at 70% CPU/memory. 5) Autoscaling buffers — ensure scale-out can happen before SLO breach (allow for pod startup time). 6) Node autoscaling — Cluster Autoscaler or Karpenter with diverse instance types. 7) Budget — RESERVED_INSTANCES for baseline + On-Demand/Spot for burst. Monitor via CloudWatch Container Insights or Prometheus. Review quarterly.
