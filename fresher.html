A. Linux Fundamentals

Foundation ⏱ ~45 min
💡 Linux is the backbone of DevOps. Every server, container, and CI runner runs Linux. Master this first.

File System Hierarchy

/ ├── bin/ # Essential user binaries (ls, cat, cp) ├── sbin/ # System binaries (ifconfig, fdisk) ├── etc/ # Configuration files (/etc/hosts, /etc/fstab) ├── home/ # User home directories ├── var/ # Variable data (logs, databases) ├── tmp/ # Temporary files (cleared on reboot) ├── usr/ # User programs (/usr/bin, /usr/lib) ├── proc/ # Virtual FS for process info (read-only) └── dev/ # Device files (disks, terminals)

File Permissions Deep Dive

# Permission format: -rwxrwxrwx  (type)(owner)(group)(others)
# r=read(4)  w=write(2)  x=execute(1)

ls -la file.txt
# -rw-r--r-- 1 user group 1234 Jan 1 10:00 file.txt

chmod 755 script.sh      # rwxr-xr-x — owner full, others read+exec
chmod 644 config.txt     # rw-r--r-- — owner read+write, others read only
chmod +x deploy.sh       # add execute for all
chmod u+rw,go-w file     # symbolic mode

chown user:group file    # change owner
chown -R user:group dir/ # recursive
bash

C. Git & Version Control

Foundation ⏱ ~35 min
🔑 Git is non-negotiable for any DevOps role. Every pipeline starts with a git push.

Core Git Commands

# ── Setup ──
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

# ── Basic workflow ──
git init                    # initialise new repo
git clone              # clone remote repo
git status                  # what's changed?
git add .                   # stage all changes
git add -p                  # stage interactively (recommended!)
git commit -m "feat: add login page"
git push origin main        # push to remote

# ── Branching ──
git checkout -b feature/login   # create + switch branch
git branch -a                   # list all branches
git merge feature/login         # merge into current branch
git rebase main                 # rebase onto main

# ── Undo ──
git restore            # discard unstaged changes
git reset HEAD         # unstage file
git revert      # create new commit that undoes change (safe)
git reset --hard HEAD~1      # DANGER: discard last commit completely

# ── Useful ──
git log --oneline --graph    # visual commit history
git stash                    # save changes temporarily
git stash pop                # restore stashed changes
git diff HEAD                # what changed since last commit
bash

Resolving Merge Conflicts

# 1. Attempt merge
git merge feature/login
# CONFLICT — Auto-merging app.py failed

# 2. Open conflicted file — it looks like this:
# <<<<<<< HEAD  (your current branch)
# login_attempts = 3
# =======       (divider)
# login_attempts = 5
# >>>>>>> feature/login  (incoming branch)

# 3. Edit file: keep the version you want, delete markers
login_attempts = 5   # keep this

# 4. Stage and commit
git add app.py
git commit -m "fix: resolve login attempts conflict"

Your First Dockerfile

# ── Simple Node.js App ──
FROM node:20-alpine          # base image (Alpine = small Linux)
WORKDIR /app                 # set working directory
COPY package*.json ./        # copy package files first (for cache)
RUN npm ci --only=production # install dependencies
COPY . .                     # copy rest of app code
EXPOSE 3000                  # document port (informational)
CMD ["node", "server.js"]    # command to start app

# ── Build and run ──
docker build -t myapp:1.0 .      # build image
docker run -d -p 3000:3000 myapp:1.0  # run container
docker run -d -p 3000:3000 \
  -e NODE_ENV=production \
  -v /data:/app/data \
  myapp:1.0                       # with env var and volume
dockerfile

version: '3.9'
services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      - DB_HOST=db
      - NODE_ENV=development
    depends_on:
      - db

      POSTGRES_USER: user

    volumes:

  pgdata:

E. CI/CD Introduction

Automation ⏱ ~35 min
💡 CI/CD automates the journey from code push to production deployment. This is the core skill that makes you a DevOps engineer.

CI/CD Pipeline Flow

Developer Push │ ▼ [ CI ] Code Checkout → Install → Lint → Test → Build → Docker Build → Push to Registry │ ▼ [ CD ] Pull Image → Deploy to Staging → Smoke Test → Deploy to Production │ ▼ [ Monitor ] Alerts if errors spike → Rollback if needed

GitHub Actions — Your First Pipeline

# .github/workflows/ci.yml
name: CI Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest    # VM that runs the job

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run linter
        run: npm run lint

      - name: Run tests
        run: npm test

      - name: Build app
        run: npm run build

  docker:
    needs: test               # runs only if test job passes
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Build Docker image
        run: docker build -t myapp:${{ github.sha }} .

      - name: Push to Docker Hub
        run: |
          echo "${{ secrets.DOCKER_PASSWORD }}" | docker login -u "${{ secrets.DOCKER_USERNAME }}" --password-stdin
          docker push myapp:${{ github.sha }}
yaml

Interview Prep — Fresher Level

Interview ⏱ ~25 min
What is DevOps and why is it important? ▼
DevOps is a culture and set of practices that combine software development (Dev) and IT operations (Ops) to shorten the development lifecycle and deliver high-quality software continuously. It's important because it breaks silos between dev and ops teams, enables faster deployments (from months to hours), improves reliability through automation, and allows rapid response to business needs. Companies like Netflix deploy hundreds of times per day using DevOps practices.
Explain the difference between rm and rm -rf in Linux ▼
rm file.txt removes a single file. rm -r dir/ removes a directory recursively. rm -f file force-removes without prompting. rm -rf dir/ combines both: recursively removes everything in the directory without any confirmation. It is one of the most dangerous commands in Linux — rm -rf / would delete your entire filesystem. Always double-check the path before running it.
What is the difference between a Docker image and a Docker container? ▼
A Docker image is a read-only blueprint/template that contains the application code, runtime, libraries, and dependencies. It's built from a Dockerfile and stored in a registry. A Docker container is a running instance of an image — it's the image brought to life with a writable layer on top. You can run many containers from the same image (like running multiple copies of your app). Think of the image as a cookie cutter and containers as the cookies.
What is CI/CD? What is the difference between CI and CD? ▼
CI (Continuous Integration): Automatically integrating code changes from multiple developers into a shared repository, running tests on every push to detect issues early. CD (Continuous Delivery): Automatically deploying code to a staging environment after CI passes — requires manual approval for production. CD (Continuous Deployment): Fully automated — every change that passes tests goes straight to production without human approval. The key distinction is the manual gate between delivery and deployment.
What is DNS and how does it work? ▼
DNS (Domain Name System) translates human-readable domain names (google.com) into IP addresses (142.250.185.78). When you type a URL: 1) Browser checks local cache → 2) OS checks /etc/hosts → 3) Query sent to configured DNS resolver (usually your ISP's) → 4) Resolver queries root servers → TLD servers (.com) → authoritative nameservers → 5) Returns IP address → 6) Browser connects to that IP. TTL (Time-To-Live) controls how long entries are cached.
Explain Git merge vs Git rebase ▼
Merge: Creates a new "merge commit" joining two branches, preserving the complete history of both branches. The history shows exactly when branches diverged and merged. Use merge for public/shared branches. Rebase: Moves (replays) your commits on top of another branch, creating a linear history as if you had started your work from the latest commit of the target branch. Cleaner history but rewrites commits. Use rebase for local feature branches before merging. Golden rule: never rebase shared/public branches.
What are environment variables and why are they used? ▼
Environment variables are key-value pairs stored outside your application code that configure the app's behavior at runtime. They're used to: 1) Keep secrets (passwords, API keys) out of code and version control. 2) Allow the same Docker image to run in dev/staging/prod with different configs. 3) Make the app "12-factor app" compliant. Examples: DATABASE_URL, NODE_ENV, API_KEY. In Docker: docker run -e NODE_ENV=production myapp. In GitHub Actions: stored as encrypted secrets.
What is the use of .gitignore? ▼
The .gitignore file tells Git which files and directories to not track or commit. This is critical for: 1) Keeping secrets safe — never commit .env files with passwords. 2) Reducing repo size — don't commit node_modules/ (thousands of files). 3) Avoiding OS clutter — ignore .DS_Store (Mac) or Thumbs.db (Windows). Common entries: node_modules/, *.log, .env, dist/, *.pyc. If you accidentally commit a file, git rm --cached filename stops tracking it.
