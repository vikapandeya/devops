File Permissions (Deep Dive)

# Permission format: [type][owner][group][other]
# Example: -rwxr-xr-- 1 ubuntu devops 2048 Apr 21 app.sh

# Breakdown:
-        # file type (- = file, d = dir, l = symlink)
rwx      # owner: read(4), write(2), execute(1) = 7
r-x      # group: read(4) + execute(1) = 5
r--      # other: read(4) = 4

# Octal: 754
chmod 754 app.sh
chmod u+x app.sh          # add execute for owner
chmod go-w sensitive.conf # remove write from group+other
chown ubuntu:devops app.sh
chown -R www-data /var/www
Never set 777 on files in production. Use least-privilege permissions. 644 for files, 755 for directories is the standard.

#!/bin/bash
set -euo pipefail

log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*"; }
die() { log "ERROR: $*" >&2; exit 1; }

check_service() {
  local service=$1
  systemctl is-active --quiet "$service" \
    && log "$service is running" \
    || die "$service is not running"
}

deploy_app() {
  local env=$1
  local services=("nginx" "postgresql" "redis")

  log "Starting deployment to $env"

  for svc in "${services[@]}"; do
    check_service "$svc"
  done

  # Pull latest code
  cd /var/www/myapp
  git pull origin main

  # Restart app
  systemctl restart myapp
  log "Deployment complete!"
}

Advanced: System Health Check Script

#!/bin/bash
set -euo pipefail

THRESHOLD_CPU=80
THRESHOLD_MEM=85
THRESHOLD_DISK=90
ALERT_EMAIL="ops@company.com"

get_cpu_usage() {
  top -bn1 | grep "Cpu(s)" | awk '{print $2}' | cut -d'%' -f1 | cut -d',' -f1
}

get_mem_usage() {
  free | grep Mem | awk '{printf "%.0f", $3/$2 * 100}'
}

get_disk_usage() {
  df -h / | awk 'NR==2 {print $5}' | tr -d '%'
}

send_alert() {
  local metric=$1 value=$2 threshold=$3
  echo "ALERT: $metric is at ${value}% (threshold: ${threshold}%)" \
    | mail -s "Server Alert: $metric High" "$ALERT_EMAIL"
}

main() {
  local cpu mem disk
  cpu=$(get_cpu_usage)
  mem=$(get_mem_usage)
  disk=$(get_disk_usage)

  echo "CPU: ${cpu}% | MEM: ${mem}% | DISK: ${disk}%"

  [[ ${cpu%.*} -gt $THRESHOLD_CPU ]] && send_alert "CPU" "$cpu" "$THRESHOLD_CPU"
  [[ $mem -gt $THRESHOLD_MEM ]]     && send_alert "Memory" "$mem" "$THRESHOLD_MEM"
  [[ $disk -gt $THRESHOLD_DISK ]]   && send_alert "Disk" "$disk" "$THRESHOLD_DISK"
}

main

Interview Questions — Linux

What is the difference between a process and a thread in Linux?
A process is an independent program with its own memory space, PID, and resources. A thread is a lightweight unit of execution within a process, sharing the same memory space. Threads are faster to create/switch between but share data, requiring synchronization (mutexes). In Linux, both are managed via clone() syscall; ps aux shows processes, ps -eLf shows threads.
How do you troubleshoot a server that is running out of disk space?
1) Run df -h to identify the full filesystem. 2) Run du -sh /var/log/* | sort -rh | head -20 to find the largest directories. 3) Check for large log files: find /var/log -size +100M. 4) Check deleted-but-open files: lsof | grep deleted — processes may hold references to deleted files. 5) Truncate logs or rotate: logrotate -f /etc/logrotate.conf. 6) Consider adding disk or archiving old data.
Explain Linux load average. What does "5.5 3.2 1.1" mean?
Load average shows the average number of processes in runnable or uninterruptible sleep state over 1, 5, and 15 minutes. "5.5 3.2 1.1" means load is increasing (1-min is highest). On a 4-core system, a load of 4.0 means 100% utilization — 5.5 means you're overloaded. Rule: if load average consistently exceeds number of CPU cores, investigate with top, vmstat, and iostat.
What happens when you run a command in Linux? (Process lifecycle)
1) Shell forks a child process via fork(). 2) Child calls exec() to load the binary. 3) Kernel creates process, assigns PID, allocates memory. 4) Process runs, makes system calls for I/O etc. 5) Process exits via exit(). 6) Parent receives SIGCHLD, calls wait() to collect exit status. If parent doesn't wait, child becomes a zombie process.
Scenario — Senior Level

SSH into a server and find it's unresponsive. How do you diagnose?

Step 1: Check if SSH daemon is running: systemctl status sshd

Step 2: Check system load: uptime or w

Step 3: Look for zombie processes or runaway processes: ps aux | awk '$8=="Z"'

Step 4: Check memory pressure: free -m and cat /proc/meminfo | grep -i swap

Step 5: Check OOM killer: dmesg | grep -i "killed process"

Step 6: Check disk full: df -h and df -i (inodes!)

Step 7: Review recent syslog: tail -200 /var/log/syslog

TCP vs UDP Deep Dive

TCP — Reliable

  • Connection-oriented (3-way handshake)
  • Guaranteed delivery & ordering
  • Flow control & congestion control
  • Used by: HTTP, SSH, FTP, SMTP
  • Higher overhead

UDP — Fast

  • Connectionless — fire and forget
  • No guarantee of delivery or order
  • Used by: DNS, video streaming, gaming
  • Lower overhead, faster
  • App handles reliability if needed

Interview Questions — Networking

What is the difference between a reverse proxy and a load balancer?
A reverse proxy sits in front of servers, receiving client requests and forwarding them to backend servers — it hides server identity and can do SSL termination, caching, compression (e.g., Nginx as reverse proxy). A load balancer specifically distributes traffic across multiple servers for availability and scalability (e.g., AWS ELB). In practice, tools like Nginx and HAProxy can do both simultaneously.
A user reports the website is slow. How do you debug it?
Systematic approach: 1) Check if issue is DNS: dig to verify resolution time. 2) Check TTFB (Time to First Byte) with curl -w "%{time_total}". 3) Check server CPU/memory: top, free. 4) Check database queries (slow query log). 5) Check network latency with traceroute. 6) Check application logs for errors. 7) Check CDN or cache hit rates. 8) Use Chrome DevTools network waterfall for frontend analysis.

Merge vs Rebase

git merge

  • Creates a merge commit
  • Preserves full history
  • Non-destructive
  • Can clutter history
  • Use for: feature → main merges
git checkout main
git merge feature/login

git rebase

  • Rewrites commit history
  • Cleaner linear history
  • Destructive if shared
  • Never rebase public branches
  • Use for: updating feature branch
git checkout feature/login
git rebase main
Golden rule: Never rebase commits that have been pushed to a shared/public branch. It rewrites history and breaks other people's work.

Interview Questions — Git

What is git cherry-pick and when would you use it?
Cherry-pick applies a specific commit from one branch to another without merging the entire branch. Use case: a critical bug fix was committed on a feature branch — you cherry-pick just that commit to main/production without merging unfinished features. Be careful: cherry-picking creates a new commit with a different SHA, which can cause issues if the original branch is later merged.
What is git reflog and when would you use it?
Reflog records every change to HEAD (even resets and rebases). It's your safety net when you think you've lost commits. Run git reflog to see the history of HEAD movements, find the commit SHA you need, then git checkout SHA or git reset --hard SHA to recover. Reflog entries expire after 90 days by default.

Jenkinsfile (Declarative Pipeline)

pipeline {
  agent any

  environment {
    APP_NAME    = 'myapp'
    DOCKER_REPO = 'myorg/myapp'
    SONAR_TOKEN = credentials('sonar-token')
  }

  stages {
    stage('Checkout') {
      steps {
        git branch: 'main', url: 'https://github.com/org/repo.git'
      }
    }

    stage('Build') {
      steps {
        sh 'mvn clean package -DskipTests'
      }
    }

    stage('Test') {
      parallel {
        stage('Unit Tests') {
          steps { sh 'mvn test' }
        }
        stage('Lint') {
          steps { sh 'npm run lint' }
        }
      }
    }

    stage('Code Quality') {
      steps {
        withSonarQubeEnv('SonarQube') {
          sh 'mvn sonar:sonar'
        }
      }
    }

    stage('Quality Gate') {
      steps {
        timeout(time: 5, unit: 'MINUTES') {
          waitForQualityGate abortPipeline: true
        }
      }
    }

    stage('Docker Build & Push') {
      steps {
        script {
          def image = docker.build("${DOCKER_REPO}:${BUILD_NUMBER}")
          docker.withRegistry('https://registry.hub.docker.com', 'dockerhub-creds') {
            image.push()
            image.push('latest')
          }
        }
      }
    }

    stage('Deploy to Staging') {
      steps {
        sh """
          kubectl set image deployment/myapp \
            myapp=${DOCKER_REPO}:${BUILD_NUMBER} \
            --namespace=staging
          kubectl rollout status deployment/myapp --namespace=staging
        """
      }
    }

    stage('Deploy to Production') {
      when {
        branch 'main'
      }
      input {
        message "Deploy to production?"
        ok "Deploy"
      }
      steps {
        sh """
          kubectl set image deployment/myapp \
            myapp=${DOCKER_REPO}:${BUILD_NUMBER} \
            --namespace=production
          kubectl rollout status deployment/myapp --namespace=production
        """
      }
    }
  }

  post {
    always {
      junit '**/target/surefire-reports/*.xml'
      archiveArtifacts artifacts: 'target/*.jar', fingerprint: true
    }
    success {
      slackSend color: 'good', message: "Build ${BUILD_NUMBER} deployed successfully"
    }
    failure {
      slackSend color: 'danger', message: "Build ${BUILD_NUMBER} FAILED"
      emailext body: '${BUILD_LOG}', subject: 'Build Failed', to: 'team@company.com'
    }
  }
}

          scan-type: 'fs'
          exit-code: '1'

    runs-on: ubuntu-latest
    needs: [test, security-scan]
    if: github.ref == 'refs/heads/main'
    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4

          registry: ${{ env.REGISTRY }}

          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}

        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  deploy:
    runs-on: ubuntu-latest
    needs: build-and-push
    environment: production  # requires manual approval in GitHub

    steps:
      - uses: actions/checkout@v4
      - name: Deploy to Kubernetes
        uses: azure/k8s-deploy@v4
        with:
          manifests: k8s/deployment.yaml
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}

Interview Questions — CI/CD

What is the difference between Continuous Integration, Continuous Delivery, and Continuous Deployment?
CI: Developers merge code frequently (multiple times/day). Each merge triggers automated build and tests. Goal: detect integration issues early.
CD (Delivery): Every successful build is automatically prepared for production release, but requires manual approval to actually deploy.
CD (Deployment): Every successful build that passes tests is automatically deployed to production without human intervention. Requires excellent test coverage and monitoring.
How do you handle secrets in CI/CD pipelines?
Never hardcode secrets. Use: 1) CI/CD platform's secret store (GitHub Secrets, GitLab CI Variables, Jenkins Credentials). 2) HashiCorp Vault integration — pipeline fetches secrets at runtime. 3) AWS Secrets Manager / Parameter Store for cloud deployments. 4) Environment-specific secrets injected at deploy time. Always audit secret access, rotate regularly, and scan code with tools like truffleHog or gitleaks to detect accidentally committed secrets.
Your pipeline takes 45 minutes to run. How do you optimize it?
1) Parallelize: run tests in matrix, separate stages concurrently. 2) Cache dependencies (node_modules, Maven .m2, pip cache). 3) Use incremental builds — only rebuild changed modules. 4) Split test suites — unit tests fast, integration tests separate. 5) Use Docker layer caching (cache-from in build). 6) Run security scans async/post-merge rather than blocking. 7) Use faster runners (more CPU, SSD). 8) Fail fast — run quickest checks first.
Scenario — Intermediate

A deployment pipeline is failing intermittently (flaky tests). What do you do?

1) Identify which test fails and how often — track failure rate over 10+ runs.

2) Check for: race conditions, test order dependencies, external service dependencies (mock them).

3) Check for: timing issues — add explicit waits, not sleep().

4) Quarantine the flaky test: move to a separate job, fix it without blocking CI.

5) Add retry logic as temporary mitigation: retry: 2 in GitLab CI or retry step in GitHub Actions.

6) Fix root cause — make tests deterministic.

Interview Questions — Docker

What is the difference between CMD and ENTRYPOINT in a Dockerfile?
CMD sets the default command and can be overridden: docker run myimage /bin/bash replaces CMD.
ENTRYPOINT sets the fixed executable that always runs — additional docker run args are passed as arguments to it.
Best practice: Use ENTRYPOINT for the main executable, CMD for default arguments. Example: ENTRYPOINT ["nginx"] CMD ["-g", "daemon off;"]. Use exec form (JSON array) to ensure proper signal handling — shell form wraps in /bin/sh -c which doesn't forward SIGTERM.
A Docker container uses too much memory and gets OOM-killed. How do you fix it?
1) Add memory limits: docker run -m 512m --memory-swap 512m myapp (--memory-swap = memory: swap disabled). 2) In compose: mem_limit: 512m. 3) Profile the app with docker stats to see actual usage. 4) Find memory leaks: exec into container, use top or heapdump tools. 5) In K8s: set resources.limits.memory. OOM kill appears in dmesg: "Killed process".
How do you reduce Docker image size in practice?
1) Multi-stage builds — build in node:20, copy output to node:20-alpine. 2) Use specific Alpine tags, not latest. 3) Combine RUN commands: RUN apt-get update && apt-get install -y pkg && rm -rf /var/lib/apt/lists/*. 4) .dockerignore to exclude unnecessary files. 5) Don't install dev tools in production stage. 6) Use distroless base images for maximum security.

Interview Questions — Kubernetes

What is the difference between a Deployment and a StatefulSet?
Deployment: For stateless apps. Pods are interchangeable — any pod can be replaced. Storage is ephemeral. No stable network identity. Scale up/down freely.
StatefulSet: For stateful apps (databases, Kafka). Each pod has stable hostname (pod-0, pod-1), stable storage (PVC per pod), ordered startup/shutdown. Used for: PostgreSQL, MongoDB, Kafka, ZooKeeper. Scale-up creates pod-N+1, scale-down removes pod-N first.
What are liveness vs readiness probes and why do both matter?
Liveness: "Is this container alive?" — If it fails, K8s kills and restarts the container. Use for: detecting deadlocks, hung processes.
Readiness: "Is this container ready to receive traffic?" — If it fails, K8s removes the pod from Service endpoints (stops sending traffic) but doesn't restart. Use for: startup time, temporary overload, dependency not ready.
Both matter: without readiness, traffic hits pods not yet ready; without liveness, hung pods stay running receiving requests they can't handle.
Explain Kubernetes resource requests vs limits.
Requests: Minimum guaranteed resources. Used by scheduler to decide which node can fit the pod. If you request 256Mi, the node must have 256Mi available.
Limits: Maximum allowed. CPU limit = throttling (slowed, not killed). Memory limit = OOMKill (process killed).
Best practice: Set both. Requests should be your normal usage, limits at 2-3x requests. If requests > limits, pod won't schedule. If no limits set, pod can consume all node resources.

Interview Questions — Cloud

What is the difference between Security Groups and NACLs in AWS?
Security Groups: Instance-level, stateful (if you allow inbound on 80, return traffic is automatic), only allow rules, no deny rules. Changes take effect immediately.
NACLs: Subnet-level, stateless (must explicitly allow return traffic), support both allow and deny rules, rules evaluated in number order. Good for blocking specific IPs subnet-wide.
Use both: NACLs for coarse-grained subnet protection, Security Groups for fine-grained instance control.
How does AWS Auto Scaling decide when to scale in/out?
Based on CloudWatch alarms tied to metrics (CPU, memory, custom metrics, SQS queue depth, etc.). Target tracking is simplest — set CPU target at 60%, ASG maintains it by adding/removing instances. Scale-out is aggressive (fast); scale-in has a cooldown period (default 300s) to prevent thrashing. Warm-up period ensures new instances aren't counted in metrics until they're ready. Always use multiple AZs for high availability.

Interview Questions — Terraform

What happens if two team members run terraform apply simultaneously?
State corruption can occur — both may read the same state and apply conflicting changes. Solution: use DynamoDB state locking with S3 backend. When one apply runs, it acquires a lock in DynamoDB. The second apply waits (or fails with lock error). If a lock gets stuck (apply crashed), manually remove it: terraform force-unlock LOCK_ID. Always use remote backend with locking in teams — never use local state files.
What is terraform taint and when do you use it?
terraform taint resource.name marks a resource for destruction and recreation on the next apply — even if its configuration hasn't changed. Use when: a resource is in a bad state (misconfigured EC2, corrupted DB), a resource needs to be replaced without changing config, you want to force a new key pair or certificate. In Terraform 1.0+, preferred approach is terraform apply -replace="resource.name" which does the same in one step.

Interview Questions — Monitoring

What is the difference between metrics, logs, and traces?
Metrics: Numerical time-series data — CPU%, request count, latency p99. Low storage cost. Good for alerting on known failure modes.
Logs: Structured or unstructured text events — error messages, access logs. High storage cost. Good for debugging specific incidents.
Traces: End-to-end journey of a request through distributed services — shows where time is spent across microservices. Tools: Jaeger, Zipkin, AWS X-Ray. Together they provide full observability.
Production alert firing at 3AM: CPU at 95%. Walk me through your response.
1) Acknowledge alert (prevent escalation). 2) Open Grafana — check CPU trend (sudden spike or gradual?). 3) top / kubectl top pods — identify which process/pod is consuming CPU. 4) Check logs for errors or unusual patterns: kubectl logs -f pod-name. 5) Check if traffic increased: Grafana request rate dashboard. 6) If runaway process: restart pod/service (mitigate first, fix later). 7) If traffic spike: trigger manual scaling. 8) Post-incident: root cause analysis, adjust alert thresholds, add runbook.

High Availability Design Principles

  • Eliminate single points of failure: Every component should have at least 2 instances

How do you achieve database zero-downtime migrations in CI/CD?
Use expand-contract (parallel change) pattern: 1) Expand: Deploy migration that adds new column as nullable (backward compatible). Run alongside old code. 2) Migrate: Backfill existing rows. 3) Contract: Deploy new application code that uses new column. 4) Cleanup: Remove old column in a later migration. Never: rename columns in one step, add NOT NULL without default, drop columns while old code still runs. Use tools like Flyway or Liquibase with versioned migrations in the repo.

Beginner Questions (0–1 year)

What is DevOps and why is it important?
DevOps is a culture and set of practices that combines software development (Dev) and IT operations (Ops) to shorten the development lifecycle and deliver high quality software continuously. It breaks down the traditional wall between developers (write code) and operations (run infrastructure). Key benefits: faster releases (from months to hours), more reliable systems, quicker recovery from failures, and better collaboration. Companies like Netflix, Amazon, and Google release code thousands of times per day using DevOps principles.
What is the difference between Docker and a Virtual Machine?
VMs virtualize hardware — each VM has a full OS (GBs of overhead), managed by a hypervisor. Boot time: minutes. Containers share the host OS kernel — only the app and its dependencies are packaged. Size: MBs. Boot time: milliseconds. Containers are more lightweight and faster, but less isolated (share kernel). VMs offer stronger isolation. In practice, containers run inside VMs in production (EC2 running Docker or K8s).
Explain what version control is and why it matters.
Version control (Git) tracks every change to code — who made it, when, and why. Benefits: 1) History — revert to any previous version. 2) Collaboration — multiple developers work simultaneously without overwriting each other. 3) Branching — experiment safely without affecting main codebase. 4) Audit trail — required for compliance in many industries. Without it, teams email zip files of code, accidentally overwrite each other's work, and can't recover from bugs.
What is SSH and how does key-based authentication work?
SSH (Secure Shell) is an encrypted protocol for remote server access. Key-based auth uses asymmetric cryptography: you have a private key (never leave your machine) and a public key (placed on server in ~/.ssh/authorized_keys). When connecting, the server sends a challenge encrypted with your public key — only your private key can decrypt it, proving your identity without sending a password over the network. More secure than passwords: no brute-force risk, no phishing. Generate with: ssh-keygen -t ed25519 -C "email@example.com".

Intermediate Questions (2–4 years)

How would you implement zero-downtime deployments?
Multiple approaches: 1) Rolling update (K8s default): replace pods one at a time, keep minAvailable=1. 2) Blue-Green: run two identical environments, switch load balancer. Instant rollback by switching back. 3) Canary: route 5% traffic to new version, monitor error rates, gradually increase. 4) Feature flags: deploy code with feature disabled, enable for users gradually. Key requirements for all: health checks, graceful shutdown (SIGTERM handling, drain connections), session affinity or stateless design, database migration backward compatibility.
Explain Kubernetes networking — how does a request reach a Pod?
Request flow: 1) Client → LoadBalancer (cloud LB, e.g. ELB) 2) → Ingress Controller (nginx-ingress pod) based on host/path rules 3) → Service (ClusterIP) via iptables/IPVS rules managed by kube-proxy 4) → Pod (selected by service's label selector). Key: every Pod gets a unique cluster-IP. Services provide stable DNS names and load-balance across pods. kube-proxy on each node programs iptables rules that intercept service traffic and forward to pod IPs. Overlay networks (Calico, Flannel) enable cross-node pod communication.
What is Terraform state and why must it be stored remotely?
Terraform state is a JSON file (.tfstate) that maps your HCL resources to real cloud resources. Terraform reads state to know what exists before computing a plan. Remote storage (S3 + DynamoDB) is critical for teams because: 1) Shared source of truth — all team members and CI/CD see the same state. 2) State locking (DynamoDB) — prevents two concurrent applies from corrupting state. 3) Backup — S3 versioning lets you recover from corrupted state. 4) Encryption — state contains sensitive values (passwords, keys). Local state in git = accidental secret exposure + merge conflicts.
How do you monitor a microservices application?
Three pillars: 1) Metrics (Prometheus): RED method — Rate (requests/sec), Errors (error rate), Duration (latency p50/p99) per service. Alert on SLO violations. 2) Logs (ELK/Loki): Structured JSON logs with correlation IDs. Centralize all logs. Alert on error patterns. 3) Traces (Jaeger/Zipkin): Distributed tracing — follow a request across 10+ microservices. Find where latency comes from. Also: synthetic monitoring (uptime checks), dashboards in Grafana, on-call rotation with PagerDuty. Use SLOs (Service Level Objectives) to define acceptable reliability.

Advanced / Senior Questions (5+ years)

Your team is deploying 50+ times per day. How do you maintain stability?
1) Automated testing at every layer — unit (fast), integration, e2e, contract tests. 2) Progressive delivery: every deploy starts as canary (1% → 10% → 100%). 3) Automated rollback: if error rate spikes >1% or latency doubles during deploy, automated system rolls back. 4) Feature flags: decouple deploy from release — code ships dark, feature enabled independently. 5) Observability: real-time dashboards showing golden signals during every deploy. 6) Immutable infrastructure: each deploy creates new instances, never modify in place. 7) Database migrations backward-compatible always. 8) Chaos engineering: intentional failure injection to validate resilience.
How would you design a multi-region disaster recovery strategy?
Depends on RTO/RPO requirements: 1) Active-Passive (Pilot Light): minimal infrastructure in DR region, scale up on disaster. RPO: hours. RTO: hours. Cheapest. 2) Warm Standby: reduced-scale running environment in DR, scale up fast. RPO: minutes. RTO: minutes. 3) Active-Active: traffic served from both regions simultaneously. RPO: near-zero. RTO: seconds. Most expensive. Key components: Route 53 health checks + latency routing, data replication (RDS Read Replica promote, DynamoDB Global Tables, S3 Cross-Region Replication), regular DR drills (monthly failover tests), runbooks for each failure scenario.
How do you handle secrets rotation without downtime?
1) Use dynamic secrets (HashiCorp Vault): application requests credential with TTL, Vault creates/rotates automatically. 2) Dual-credential rotation: step 1 create new credential, step 2 update app to accept both, step 3 rotate all apps to new credential, step 4 revoke old. 3) External Secrets Operator in K8s: Vault/AWS Secrets Manager → K8s Secret, triggers pod rolling restart on change. 4) Application must handle credential refresh without restart (connection pool reconnect). 5) Never hard-code secrets in images, env vars in plain text, or config files checked into git.

Project 2 — CI/CD Pipeline with GitHub Actions + Docker

Level: Intermediate Time: 4-6 hours Tools: GitHub Actions, Docker, EC2

Architecture

GitHub Push → GitHub Actions → Build & Test → Docker Build → ECR/DockerHub → Deploy to EC2

Full Pipeline Setup

# .github/workflows/main.yml
name: Build and Deploy

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with: { node-version: '20', cache: 'npm' }
      - run: npm ci
      - run: npm test

  build:
    needs: test
    runs-on: ubuntu-latest
    outputs:
      image-tag: ${{ steps.meta.outputs.tags }}
    steps:
      - uses: actions/checkout@v4

      - name: Login to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Generate tags
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=sha,prefix=sha-
            type=raw,value=latest,enable=${{ github.ref == 'refs/heads/main' }}

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: ${{ github.ref == 'refs/heads/main' }}
          tags: ${{ steps.meta.outputs.tags }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  deploy:
    needs: build
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - name: Deploy to EC2
        uses: appleboy/ssh-action@v1.0.3
        with:
          host: ${{ secrets.EC2_HOST }}
          username: ubuntu
          key: ${{ secrets.EC2_SSH_KEY }}
          script: |
            echo ${{ secrets.GITHUB_TOKEN }} | \
              docker login ghcr.io -u ${{ github.actor }} --password-stdin
            docker pull ghcr.io/${{ github.repository }}:latest
            docker stop myapp || true
            docker rm myapp || true
            docker run -d \
              --name myapp \
              -p 80:3000 \
              --restart unless-stopped \
              -e NODE_ENV=production \
              ghcr.io/${{ github.repository }}:latest
            docker image prune -f
Expected outcome: Every push to main builds, tests, and deploys automatically within 3-5 minutes.

Deploy a production website across two EC2 instances: one running the application (Nginx + PHP-FPM) accessible via a public ALB, and one running MySQL in a private subnet — both wired together behind an Application Load Balancer.

Architecture Overview

LEMP = Linux + ENginx + MySQL + PHP. This is the standard production stack for WordPress, Laravel, and custom PHP apps.

Prerequisites

  • AWS account with IAM user having EC2, VPC, ELB full access
  • AWS CLI configured: aws configure
  • An EC2 Key Pair created in your target region
  • Domain name (optional — for HTTPS via ACM)

Step 1 — Create VPC & Subnets

# Create VPC
aws ec2 create-vpc --cidr-block 10.0.0.0/16 \
  --tag-specifications 'ResourceType=vpc,Tags=[{Key=Name,Value=lemp-vpc}]'
# Note the VpcId returned, e.g. vpc-0abc123

VPC_ID=vpc-0abc123   # replace with your VpcId

# Enable DNS hostnames (required for EC2 hostname resolution)
aws ec2 modify-vpc-attribute --vpc-id $VPC_ID \
  --enable-dns-hostnames '{"Value":true}'
aws ec2 modify-vpc-attribute --vpc-id $VPC_ID \
  --enable-dns-support '{"Value":true}'

# Create Internet Gateway and attach
IGW_ID=$(aws ec2 create-internet-gateway \
  --tag-specifications 'ResourceType=internet-gateway,Tags=[{Key=Name,Value=lemp-igw}]' \
  --query 'InternetGateway.InternetGatewayId' --output text)
aws ec2 attach-internet-gateway --internet-gateway-id $IGW_ID --vpc-id $VPC_ID

# Public subnet (App server lives here)
PUB_SUBNET=$(aws ec2 create-subnet \
  --vpc-id $VPC_ID \
  --cidr-block 10.0.1.0/24 \
  --availability-zone us-east-1a \
  --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=lemp-public}]' \
  --query 'Subnet.SubnetId' --output text)

# Enable auto-assign public IP on public subnet
aws ec2 modify-subnet-attribute --subnet-id $PUB_SUBNET \
  --map-public-ip-on-launch

# Private subnet (MySQL lives here)
PRIV_SUBNET=$(aws ec2 create-subnet \
  --vpc-id $VPC_ID \
  --cidr-block 10.0.2.0/24 \
  --availability-zone us-east-1b \
  --tag-specifications 'ResourceType=subnet,Tags=[{Key=Name,Value=lemp-private}]' \
  --query 'Subnet.SubnetId' --output text)

# Public route table → Internet Gateway
RT_ID=$(aws ec2 create-route-table --vpc-id $VPC_ID \
  --query 'RouteTable.RouteTableId' --output text)
aws ec2 create-route --route-table-id $RT_ID \
  --destination-cidr-block 0.0.0.0/0 --gateway-id $IGW_ID
aws ec2 associate-route-table --route-table-id $RT_ID \
  --subnet-id $PUB_SUBNET

echo "VPC=$VPC_ID  PUB=$PUB_SUBNET  PRIV=$PRIV_SUBNET  IGW=$IGW_ID"

Interview Questions — This Setup

Why is the MySQL instance in a private subnet with no public IP?
Security principle of least exposure — the database should never be directly reachable from the internet. A public IP means anyone can attempt connections on port 3306 (brute-force, credential stuffing). In the private subnet with DB security group allowing port 3306 only from the App security group, only the application server can reach MySQL — even if an attacker compromises another service, they cannot reach the DB directly. Defense in depth: network isolation is the first layer, strong passwords and IAM are additional layers.
Why use ALB instead of pointing DNS directly to the App EC2 public IP?
1) Scalability: ALB can route to multiple app servers (Auto Scaling Group) — direct IP only reaches one. 2) Health checks: ALB stops sending traffic to unhealthy instances automatically. 3) SSL termination: ACM certificate attached to ALB handles HTTPS — instances serve plain HTTP internally. 4) Fixed endpoint: EC2 public IPs change on restart; ALB DNS name is stable. 5) Security: App SG allows port 80 only from ALB SG — EC2 is never directly internet-accessible even though it's in a public subnet.

Step 1 — VPC with Full Subnet Layout

#!/bin/bash
set -e

REGION="us-east-1"
PROJECT="prod"

# Create VPC
VPC_ID=$(aws ec2 create-vpc \
  --cidr-block 10.0.0.0/16 --region $REGION \
  --tag-specifications "ResourceType=vpc,Tags=[{Key=Name,Value=${PROJECT}-vpc}]" \
  --query 'Vpc.VpcId' --output text)

aws ec2 modify-vpc-attribute --vpc-id $VPC_ID --enable-dns-hostnames '{"Value":true}'
aws ec2 modify-vpc-attribute --vpc-id $VPC_ID --enable-dns-support '{"Value":true}'

# Internet Gateway
IGW_ID=$(aws ec2 create-internet-gateway \
  --tag-specifications "ResourceType=internet-gateway,Tags=[{Key=Name,Value=${PROJECT}-igw}]" \
  --query 'InternetGateway.InternetGatewayId' --output text)
aws ec2 attach-internet-gateway --internet-gateway-id $IGW_ID --vpc-id $VPC_ID

# ── Public Subnets ──────────────────────────────────────────────────────────
PUB_A=$(aws ec2 create-subnet --vpc-id $VPC_ID \
  --cidr-block 10.0.1.0/24 --availability-zone ${REGION}a \
  --tag-specifications "ResourceType=subnet,Tags=[{Key=Name,Value=${PROJECT}-public-a}]" \
  --query 'Subnet.SubnetId' --output text)

PUB_B=$(aws ec2 create-subnet --vpc-id $VPC_ID \
  --cidr-block 10.0.2.0/24 --availability-zone ${REGION}b \
  --tag-specifications "ResourceType=subnet,Tags=[{Key=Name,Value=${PROJECT}-public-b}]" \
  --query 'Subnet.SubnetId' --output text)

aws ec2 modify-subnet-attribute --subnet-id $PUB_A --map-public-ip-on-launch
aws ec2 modify-subnet-attribute --subnet-id $PUB_B --map-public-ip-on-launch

# ── Private App Subnets ──────────────────────────────────────────────────────
APP_A=$(aws ec2 create-subnet --vpc-id $VPC_ID \
  --cidr-block 10.0.11.0/24 --availability-zone ${REGION}a \
  --tag-specifications "ResourceType=subnet,Tags=[{Key=Name,Value=${PROJECT}-app-a}]" \
  --query 'Subnet.SubnetId' --output text)

APP_B=$(aws ec2 create-subnet --vpc-id $VPC_ID \
  --cidr-block 10.0.12.0/24 --availability-zone ${REGION}b \
  --tag-specifications "ResourceType=subnet,Tags=[{Key=Name,Value=${PROJECT}-app-b}]" \
  --query 'Subnet.SubnetId' --output text)

# ── Private DB Subnets ───────────────────────────────────────────────────────
DB_A=$(aws ec2 create-subnet --vpc-id $VPC_ID \
  --cidr-block 10.0.21.0/24 --availability-zone ${REGION}a \
  --tag-specifications "ResourceType=subnet,Tags=[{Key=Name,Value=${PROJECT}-db-a}]" \
  --query 'Subnet.SubnetId' --output text)

DB_B=$(aws ec2 create-subnet --vpc-id $VPC_ID \
  --cidr-block 10.0.22.0/24 --availability-zone ${REGION}b \
  --tag-specifications "ResourceType=subnet,Tags=[{Key=Name,Value=${PROJECT}-db-b}]" \
  --query 'Subnet.SubnetId' --output text)

echo "Subnets created: PUB_A=$PUB_A PUB_B=$PUB_B APP_A=$APP_A APP_B=$APP_B DB_A=$DB_A DB_B=$DB_B"

Interview Questions — This Setup

What is a Bastion host and why do you use it instead of direct SSH?
A Bastion (jump host) is a hardened EC2 instance in a public subnet whose sole purpose is to be an SSH entry point. Benefits: 1) Single audit point — all admin access goes through one server, making it easy to log and monitor. 2) Reduced attack surface — private servers have no public IPs, so they can't be port-scanned or brute-forced from the internet. 3) Fine-grained control — you whitelist only specific admin IPs on the Bastion SG. 4) Compliance — many standards (PCI-DSS, SOC2) require isolated admin access paths. Alternative: AWS Systems Manager Session Manager removes the need for a Bastion entirely (no SSH keys, no open ports, full audit trail).
Why have NAT Gateways if private servers can't be accessed from internet?
NAT Gateway provides outbound-only internet access for private instances. Private servers still need to reach the internet for: installing OS updates (dnf update), pulling Docker images, downloading application packages, calling external APIs (payment gateways, SMS services), sending CloudWatch metrics. Without NAT Gateway, private instances are completely isolated — even dnf update -y would fail. NAT Gateway allows outbound traffic but blocks all unsolicited inbound connections (it does network address translation, not port forwarding).
How would you make this architecture more resilient against AZ failures?
The architecture already handles AZ failures if set up correctly: 1) ALB is multi-AZ — if AZ-a fails, it routes 100% to App-2 in AZ-b automatically. 2) Two NAT Gateways (one per AZ) — each private route table routes through its AZ's NAT. 3) MySQL becomes the single point of failure — fix with: RDS Multi-AZ (synchronous standby in AZ-b, automatic failover in ~60s), or MySQL with read replica promoted manually. Also add Auto Scaling Group across both app subnets — replaces failed instances automatically. With this, the architecture survives full AZ loss with zero manual intervention.
How do you deploy a code update to both app servers without downtime?
Rolling deployment: 1) Deregister App-1 from ALB target group (ALB stops sending traffic to it, existing connections drain). 2) SSH via Bastion to App-1, pull new code, restart services, run smoke test. 3) Re-register App-1 in ALB, wait for health check to pass. 4) Repeat for App-2. Zero downtime because App-2 handles all traffic during App-1 update. Automate this with: an Ansible playbook with serial: 1 (update one at a time), or CodeDeploy with in-place deployment and ALB integration (handles deregistration/registration automatically).

Interview Questions — Cloudflare + Terraform + ALB

Why lock the ALB security group to Cloudflare IP ranges only?
Without this, anyone who discovers your ALB DNS name (via DNS history, certificate transparency logs, or scanning) can bypass Cloudflare entirely — hitting your origin directly and bypassing WAF, rate limiting, and DDoS protection. By restricting ALB ingress to Cloudflare's published IP ranges, the ALB refuses all non-Cloudflare traffic. Cloudflare also adds the CF-Connecting-IP header with the real client IP so your app can log it. Update the IP list periodically — Cloudflare publishes changes at cloudflare.com/ips. The Terraform variable makes this a one-line update + apply.
What is Cloudflare Full-Strict SSL mode and why is it important?
Flexible: Cloudflare ↔ Origin is HTTP (unencrypted) — dangerous.
Full: Cloudflare ↔ Origin is HTTPS but accepts self-signed certs — vulnerable to MITM between CF and ALB.
Full-Strict: Cloudflare ↔ Origin is HTTPS and validates the cert — requires a valid cert on the ALB (ACM cert). This is the only mode that gives true end-to-end encryption. In our setup: user ↔ CF = TLS (CF cert), CF ↔ ALB = TLS (ACM cert). Without Full-Strict, you have a false sense of security.
How do you manage Cloudflare API tokens securely in Terraform CI/CD?
Never hardcode in .tf files or tfvars committed to git. Use: 1) GitHub Actions Secrets → passed as TF_VAR_cloudflare_api_token environment variable. 2) HashiCorp Vault → CI fetches token at runtime. 3) AWS Secrets Manager → fetch in pipeline before terraform apply. 4) Mark the variable sensitive = true in Terraform so it never appears in plan output or state diffs. 5) Create a scoped API token in Cloudflare (Zone:DNS:Edit + Zone:Firewall:Edit only — not the global API key). Rotate every 90 days.

Interview Questions — Kubernetes Production

What is GitOps and how does ArgoCD implement it?
GitOps is a practice where Git is the single source of truth for both application code and infrastructure state. Any change to the cluster must be made via a git commit — never direct kubectl. ArgoCD implements this by continuously watching a git repository and comparing its state against the live cluster. If they diverge (someone ran kubectl manually, or a pod crashed), ArgoCD either alerts or auto-syncs back to the git state (selfHeal). Benefits: full audit trail (git log = deployment history), easy rollback (git revert), PR-based approval for production changes, and disaster recovery (re-apply the git repo to any cluster and it self-assembles).
Why use Pod Disruption Budgets and when do they activate?
PDBs prevent Kubernetes from removing too many pods simultaneously during voluntary disruptions — node drains (kubectl drain), cluster upgrades, or Cluster Autoscaler scale-down. Without PDB: draining a node could take down all 3 replicas if they happen to be on that node, causing outage. With minAvailable: 67% on a 3-replica deployment: at most 1 pod can be evicted at a time, always keeping 2 running. PDBs do NOT apply to involuntary disruptions (node hardware failure, OOMKill). They're essential for safe cluster upgrades and maintenance windows.
How does IRSA (IAM Roles for Service Accounts) work and why is it better than node-level IAM roles?
IRSA maps a Kubernetes ServiceAccount to an AWS IAM Role using OIDC federation. The pod's service account token is exchanged for temporary AWS credentials via AWS STS — no static access keys stored anywhere. Why better than node IAM roles: with node roles, ALL pods on that node inherit the role's permissions (a compromised pod gets all node permissions). With IRSA, each pod/service gets its own minimal IAM role — blast radius of a compromise is limited to exactly what that service needs. Setup: annotate the K8s ServiceAccount with eks.amazonaws.com/role-arn, configure the IAM role trust policy to allow the OIDC provider to assume it.
A production deployment is stuck — 0 of 3 pods running after a Helm upgrade. How do you debug and recover?
1) kubectl get pods -n production — check pod status (Pending? ImagePullBackOff? CrashLoop?). 2) kubectl describe pod <pod> — read the Events section for root cause. 3) kubectl logs <pod> --previous — previous container logs. 4) kubectl get events -n production --sort-by=.lastTimestamp — cluster events. 5) If it's a bad image: helm rollback myapp 1 -n production — rolls back to previous Helm revision instantly. 6) If it's a config issue: fix values, run helm upgrade again. 7) After recovery: check why CI let a bad image through — add better smoke tests in the promote workflow.
